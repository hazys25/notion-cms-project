# Claude 명령어: Error Checking

Playwright(헤드리스 브라우저)로 개발 서버를 실제로 띄워 **주요 페이지를 순회하며 런타임 에러를 수집**하고,
수집된 에러를 **즉시 해결 가능한 것**과 **즉시 해결이 어려운 것**으로 분류해 정리합니다.

## 사용법

```
/error-checking
```

## 무엇을 점검하나 (수집 대상)

브라우저에서 페이지를 여는 동안 발생하는 아래 신호를 모두 수집합니다.

- **콘솔 에러/경고**: `console.error`, `console.warn` 출력
- **페이지 예외(pageerror)**: 처리되지 않은 JavaScript 런타임 예외
- **요청 실패(requestfailed)**: 끊긴 네트워크 요청, 404/500 등 비정상 응답
- **하이드레이션 불일치**: Next.js 서버/클라이언트 렌더링 불일치 경고
- **빌드/타입 에러**: 점검 전 `pnpm lint` + `pnpm build` 로 정적 에러도 함께 확인

## 프로세스

### 1. 사전 준비 — Playwright 확인

1. `playwright` 설치 여부를 확인합니다. (`package.json` devDependencies 또는 `node_modules/playwright`)
2. 설치되어 있지 않으면 사용자에게 설치 동의를 구한 뒤 실행합니다.
   ```bash
   pnpm add -D playwright
   pnpm exec playwright install chromium   # 브라우저 바이너리 설치
   ```
   > 설치 주의사항(corepack/PATH)은 사용자 메모리 `pnpm-setup.md` 를 따릅니다.

### 2. 정적 에러 점검 (서버 구동 전)

- `pnpm lint` 실행 → ESLint 에러/경고 수집
- `pnpm build` 실행 → 타입 에러 및 빌드 실패 수집
- 이 단계에서 막히는 에러는 브라우저를 띄우기 전에 먼저 분류 목록에 기록합니다.

### 3. 동적 에러 점검 (Playwright 구동)

1. 개발 서버를 백그라운드로 실행합니다. (`pnpm dev`, 기본 `http://localhost:3000`)
2. 서버가 응답할 때까지 대기한 뒤, 아래 점검 스크립트를 실행합니다.
3. **점검 대상 경로**: 라우트를 자동 탐색하여 구성합니다.
   - `src/app/**/page.tsx` 를 Glob 으로 찾아 실제 존재하는 경로만 방문
   - 인증 미들웨어로 보호되는 경로는 `/login` 선행 처리(필요 시 env 계정으로 로그인) 후 방문
   - 최소 점검 경로: `/login`, `/` (로그인 후)
4. 각 경로에서 위 "수집 대상" 신호를 모읍니다.

> 점검 스크립트는 임시 파일(`scripts/error-check.mjs` 등)로 생성해 실행하고, 점검 후 정리합니다.
> Playwright MCP 서버가 연결되어 있다면 스크립트 대신 MCP 브라우저 도구를 사용해도 됩니다.

### 4. 에러 분류 및 보고

수집한 모든 에러를 아래 두 그룹으로 **명확히 구분**해 나열합니다.

#### ✅ 즉시 해결 가능 (Auto-fixable)

원인이 코드 내부에 있고, 안전하게 수정 가능한 항목입니다. 예:

- 잘못된/누락된 import 경로, 오타
- 누락된 `"use client"` 지시어, 잘못된 prop 타입
- 잘못된 링크(`href`)로 인한 404
- Lint 자동 수정(`--fix`)으로 해결되는 항목
- 명백한 하이드레이션 불일치(예: 서버/클라 분기 누락)

각 항목에 대해 **원인 → 해결 방법 → 대상 파일·라인**을 함께 제시합니다.

#### ⚠️ 즉시 해결 불가 (Needs decision / 외부 요인)

추가 정보·결정·외부 환경이 필요한 항목입니다. 예:

- 환경변수 누락(`SESSION_SECRET`, Supabase 키 등) — 값은 사용자만 제공 가능
- 외부 서비스(Supabase 등) 연결 실패, 네트워크/권한 문제
- 설계 결정이 필요한 동작(어떤 동작이 의도된 것인지 확인 필요)
- 외부 라이브러리 내부 경고/버그
- 재현이 불안정해 원인 특정이 어려운 항목

각 항목에 대해 **현상 → 추정 원인 → 사용자에게 필요한 결정/정보**를 제시합니다.

### 5. 마무리

- 백그라운드 개발 서버를 종료합니다.
- 생성한 임시 점검 스크립트를 정리합니다.
- **요약 표**(에러 개수, 해결 가능 N건 / 불가 M건)로 마무리합니다.
- 사용자가 원하면 "즉시 해결 가능" 항목을 실제로 수정할지 물어봅니다. (자동 수정하지 않음 — 동의 후 진행)

## 출력 예시

```
🔎 에러 점검 결과 (http://localhost:3000)

[정적] lint ✅ 통과 / build ✅ 통과
[동적] 방문 경로 2개 (/login, /) — 콘솔 에러 3건, 요청 실패 1건

✅ 즉시 해결 가능 (2건)
1. theme-toggle import 경로 오류
   - 원인: '@/components/common/theme-toggl' (오타)
   - 해결: '@/components/common/theme-toggle' 로 수정
   - 위치: src/app/page.tsx:12

⚠️ 즉시 해결 불가 (2건)
1. Supabase 요청 실패 (401)
   - 현상: anon 키 인증 거부
   - 추정 원인: .env.local 의 NEXT_PUBLIC_SUPABASE_ANON_KEY 미설정/만료
   - 필요: 사용자가 유효한 키 제공 필요

수정을 진행할까요? (즉시 해결 가능 항목 2건)
```

## 주의사항

- 점검은 **읽기/분석 위주**로 진행하며, 코드 수정은 사용자 동의 후에만 합니다.
- 개발 서버는 점검 종료 시 반드시 함께 종료합니다(좀비 프로세스 방지).
- 환경변수 누락 에러는 "즉시 해결 불가"로 분류합니다(값은 사용자만 제공 가능).
- 분류가 애매한 항목은 임의 판단하지 말고 "즉시 해결 불가"에 넣고 사용자에게 확인을 요청합니다.
